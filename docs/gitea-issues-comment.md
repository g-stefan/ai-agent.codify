# Gitea Issues Comment Utility (`gitea-issues-comment.py`)

## Description
The `gitea-issues-comment.py` script is designed to automate the process of posting comments to a specific Gitea issue. It reads an `issue.json` file to locate the target issue (via the API `url` attribute) and a `work.json` file (typically generated by `agent-work-to-json.py`) to extract the comment body. 

Using the `requests` library, it sends an authenticated `POST` request to the Gitea API, appending the `"comment"` payload from the work file to the issue thread. This is primarily used by automated agents to provide feedback, summaries, or reports directly on the issue ticket they are resolving.

## Command-Line Options

This script utilizes `argparse` to handle its execution arguments:

| Argument | Type | Description |
| :--- | :--- | :--- |
| `-i`, `--issue` | *Required* | Path to the `issue.json` file containing the metadata (including the API `url`) of the target issue. |
| `-w`, `--work` | *Required* | Path to the `work.json` file containing the `comment` body to be posted. |
| `-t`, `--token` | *Optional* | Your Gitea API access token. If omitted, the script falls back to reading the `GITEA_TOKEN` environment variable. |
| `-h`, `--help` | *Flag* | Shows the help message and exits. |

## Prerequisites
This script requires the `requests` library to handle HTTP requests:
```sh
pip install requests
```

## Exit Codes
*   **`0` (Success)**: The comment was successfully posted to the issue, or the script exited cleanly because the provided comment body was empty (skipping the post).
*   **`1` (Error)**: The script failed. This can happen if required files are missing, the JSON format is invalid (e.g., missing `url` or `comment` fields), the API token is missing, or the Gitea server returned a non-201 HTTP status code.

## Examples

### 1. Basic Execution (Using Environment Variable)
Assuming `GITEA_TOKEN` is exported in your environment, and your JSON files are in the current directory:
```sh
python gitea-issues-comment.py -i issue.json -w work.json
```

### 2. Passing the Token Explicitly
If you need to supply the token securely via the command line (e.g., passed down from a secret manager in a CI pipeline):
```sh
python gitea-issues-comment.py --issue work.read/issue.json --work work.write/work.json --token "my_secret_token_123"
```

### 3. Understanding the Expected Inputs
The script expects the `issue.json` to be an array containing at least one issue object with an API URL:
```json
[
  {
    "number": 42,
    "title": "Fix the parser bug",
    "url": "https://gitea.example.com/api/v1/repos/org/repo/issues/42"
  }
]
```
And the `work.json` file to contain a `comment` attribute:
```json
{
  "title": "Parser fix summary",
  "comment": "I have successfully identified the bug and committed the fix to the branch."
}
```
